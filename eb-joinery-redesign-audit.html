Category 01

UI & UX

Strong

Typography pairing is confident and distinctive

Archivo + Space Grotesk is a well-matched pairing — both functional and characterful without being generic. The large display headings with tight letter-spacing (letter-spacing:-0.04em) at clamp(3.5rem, 7vw, 7.5rem) give the hero sections genuine presence. The use of italics for visual contrast within headings is an elegant detail.

Strong

Floating pill navbar is modern and well-executed

The backdrop-filter: blur(24px) pill navbar is one of the best features of the redesign. Pinned at top: 1.5rem with position:fixed and a subtle border+shadow, it floats credibly without obscuring content. The spacing, active states, and CTA button treatment all work.

Critical

Portfolio overlay labels are invisible on mobile — hover-only UX

The .portfolio-overlay uses opacity: 0 and only reveals on :hover. On touchscreen devices there is no hover state — users never see the project title or category label. This is a fundamental usability failure for a portfolio site where the gallery is a key conversion asset.

→ Recommended fix
/* Always show overlay on mobile touch devices */ @media (hover: none) { .portfolio-overlay { opacity: 1; background: linear-gradient(to top, rgba(0,0,0,0.7) 0%, transparent 50%); } }
Critical

Hero occupies 100svh but shows no scroll cue — abandonment risk

The hero is min-height: 100svh, which means the services section is entirely below the fold with zero visual indicator that content continues. Users who don't scroll instinctively will bounce. No scroll-down arrow, animated chevron, or fading content peek is present.

→ Recommended fix

Add a subtly animated scroll indicator anchored to the bottom of the hero. A simple bouncing ↓ or chevron SVG with a CSS keyframe animation would suffice and would meaningfully reduce bounce rate.

Issue

Gradient footer direction is inverted — dark-on-dark contrast failure

The footer uses background: linear-gradient(to top, #09132B 0%, #0B1B3D 50%, #162E6B 80%, #FFFFFF 100%). This means it fades from deep navy at the bottom to white at the top. The white top edge abuts the global CTA section which is also white, creating a seamless visual bleed. But more critically, footer link text is set in rgba(255,255,255,0.7) — against a near-white background at the top of the footer, this likely fails WCAG AA contrast ratios for smaller text.

→ Recommended fix
/* Correct direction: dark at top, deepest at bottom */ footer { background: linear-gradient(to bottom, #162E6B 0%, #0B1B3D 50%, #09132B 100%); }
Issue

About image hidden on mobile — content gap on a trust-critical page

On the About page, .about-img { display: none } at ≤900px. The craftsman photo and the "12 Years" badge are both hidden on all mobile devices. For a sole trader business where personal trust is the main differentiator, removing the only human image from the page on the most common device type is a significant conversion mistake.

→ Recommended fix

Restore the image on mobile at full width above the text block. The grid collapses to 1fr already, so simply removing the display:none rule and letting the image sit as a block above the content is sufficient.

Issue

Page hero sections force 100svh on inner pages — unnecessarily tall

Every inner page (Portfolio, Contact, About, Blog, Testimonials) uses min-height: 100svh on its .page-hero. This means a visitor who taps "Portfolio" from the nav must scroll a full screen height before seeing a single project photo. The actual content is the point of the page — the hero shouldn't bury it below the fold.

→ Recommended fix
/* Inner page heroes should be content-driven, not viewport-filling */ .page-hero { min-height: 40svh; /* was: 100svh */ padding: 8rem 5vw 4rem; } @media (max-width: 900px) { .page-hero { min-height: unset; } }
Issue

Accent colour is sky-blue (#38bdf8) against white — low contrast

The primary accent --accent: #38bdf8 is used heavily for section labels, service links, and stat numbers on white or light grey backgrounds. This colour achieves a contrast ratio of approximately 2.8:1 against white — well below the WCAG AA minimum of 4.5:1 for normal text. Small-text elements like .service-link, .section-label and stats in the about block are all affected.

→ Recommended fix
/* Darken accent for on-light-background usage */ :root { --accent-on-light: #0369a1; /* contrast 5.9:1 on white */ } .section-label, .service-link, .section-title em { color: var(--accent-on-light); }
Enhancement

Stats pill in hero reads weakly — deserves more visual weight

The .hero-stats-inline pill (12+ Years · 500+ Projects · 5★) is a strong social proof element but it's styled with very low opacity background (rgba(255,255,255,0.03)) and a subtle border. Combined with the small text size, it's visually forgettable. Given these numbers are key trust signals against competing local joiners, they warrant stronger treatment.

Enhancement

Blog mini-cards lack images on mobile — list becomes text-only

The .blog-mini-img thumbnails are only 76×68px which renders very small on high-DPI screens and becomes less useful on mobile. The blog list items have no fallback visual engagement on small screens. Consider enlarging thumbnails or switching to a stacked card layout at mobile widths.

Enhancement

Contact page form has no visible success state beyond a toast

On successful form submission, the page calls showToast() and resets. The toast element (#toast) is referenced in JS but isn't present in the contact.html markup — meaning the success message silently fails. The leave-a-review.html page handles this correctly by hiding the form and showing #review-msg; the contact page should follow the same pattern.

Category 02

Mobile Optimisation

Strong

iOS input zoom prevention correctly implemented

All form inputs, textareas, and selects are explicitly set to font-size: 16px !important at mobile widths. This correctly prevents the iOS Safari automatic zoom-on-focus behaviour, which is a common oversight that destroys form UX on iPhones.

Strong

Touch targets meet 48px minimum on mobile nav

The hamburger button and mobile menu links both have explicit min-height: 48px / min-width: 48px constraints, satisfying the WCAG 2.5.5 target size guideline. This is a quality detail often missed.

Strong

Service carousel handles mobile vs desktop correctly in JS

The initServiceCarousels() function distinguishes between mobile (single crossfading image) and desktop (3-column grid) via window.innerWidth <= 768. This is the right architectural decision — serving appropriately sized images and layouts per device type rather than hiding/showing via CSS alone.

Critical

Mobile menu doesn't close when a link is tapped

The toggleMobile() function only fires on the hamburger icon. If a user taps a nav link in the open mobile menu, the overlay remains open as the page navigates (or would remain if it were an SPA). Additionally, there is no tap-outside-to-close behaviour. On a full-screen overlay menu this is a standard expectation users will immediately notice when it's missing.

→ Recommended fix
// Add to DOMContentLoaded: document.querySelectorAll('.mobile-menu a').forEach(link => { link.addEventListener('click', () => { document.getElementById('mobile-menu').classList.remove('open'); }); });
Critical

Portfolio filter buttons overflow horizontally on small screens

The .portfolio-filter uses flex-wrap: wrap which technically prevents overflow, but with 8 filter buttons at their current padding (0.6rem 1.4rem), they wrap onto 3–4 lines on a 375px screen. This takes up ~200px of vertical space before the first image is visible. No horizontal scroll alternative is offered, and there is no mobile-specific reduction in the number of visible filters.

→ Recommended fix
@media (max-width: 640px) { .portfolio-filter { overflow-x: auto; flex-wrap: nowrap; -webkit-overflow-scrolling: touch; scrollbar-width: none; padding-bottom: 0.5rem; } .portfolio-filter::-webkit-scrollbar { display: none; } .filter-btn { white-space: nowrap; flex-shrink: 0; } }
Issue

Carousel breakpoint check doesn't respond to orientation changes

The initServiceCarousels() function checks window.innerWidth <= 768 once at DOMContentLoaded. A user who loads on mobile portrait then rotates to landscape (or vice versa) will continue to see the wrong carousel layout. This is particularly noticeable on tablets.

→ Recommended fix

Use a CSS media query-based approach (matchMedia) instead of reading window.innerWidth directly, or re-run the carousel init logic on window.addEventListener('resize', debounce(initServiceCarousels, 300)).

Issue

No loading="eager" or explicit LCP image prioritisation

The hero background is a CSS gradient with no above-the-fold image, which is fine for LCP. However, the home portfolio grid renders via JavaScript and all its images use loading="lazy". On slower connections, this means the entire "Latest Projects" section — which is above the fold on most desktop resolutions — loads blank with a white flash as the masonry images trickle in.

→ Recommended fix

For the first 3 images in renderHomePortfolio(), omit loading="lazy" or explicitly add fetchpriority="high" to ensure the most visible content loads promptly.

Issue

Testimonials grid collapses to single column but cards are very tall

At mobile widths the testimonials are stacked in a single column. Each card has padding: 2.5rem plus a large decorative ::before quote mark (font-size: 7rem). This results in very deep cards — the first testimonial alone can consume the full viewport height. Consider reducing the decorative quote mark to 4rem at mobile and trimming padding to 1.5rem.

Enhancement

No sticky "Call Now" CTA on mobile — missed conversion opportunity

The phone number in the nav is hidden at ≤900px (.nav-phone { display: none }) and the mobile menu requires two taps to reach it. For a local trades business, a persistent sticky bottom bar with a single "📞 Call Now" button on mobile is one of the highest-converting UI elements possible. No such element exists anywhere on the site.

→ Recommended fix
/* Sticky mobile CTA bar */ @media (max-width: 900px) { .mobile-cta-bar { position: fixed; bottom: 0; left: 0; right: 0; background: var(--accent); display: flex; z-index: 300; padding: 0.8rem 1.5rem; gap: 0.8rem; } .mobile-cta-bar a { flex: 1; text-align: center; padding: 0.9rem; font-weight: 700; text-decoration: none; border-radius: 8px; } }
Category 03

Technical SEO

Strong

LocalBusiness schema is comprehensive and correctly structured

Every page includes a LocalBusiness JSON-LD block with telephone, email, priceRange, openingHours, and a well-populated areaServed array covering all key target locations. The testimonials page correctly adds an AggregateRating and individual Review objects. This is among the strongest local SEO schema implementations you'll find on a trades website.

Strong

BreadcrumbList schema present on key pages

Portfolio, Contact, and Blog pages include BreadcrumbList schema. This improves the appearance of search results with breadcrumb display in SERPs and helps Google understand site hierarchy.

Strong

Canonical tags on every page

All 13 pages include a correctly formatted <link rel="canonical"> pointing to the full HTTPS URL. This prevents duplicate content issues and ensures link equity is consolidated correctly.

Critical

Portfolio content is entirely JavaScript-rendered — invisible to crawlers

The entire project gallery is built by renderPortfolio() and renderHomePortfolio() at runtime. The HTML for every portfolio image, title, and category is injected via innerHTML on DOMContentLoaded. While Googlebot does execute JavaScript, it processes JS-rendered content in a second wave with delays. More critically, the 500+ project titles and category information are never visible to crawlers that don't execute JS (Bing secondary crawl, social media scrapers, etc.).

→ Recommended fix

Pre-render at least the first 12 portfolio items as static HTML in the page source. The JavaScript can still populate additional items dynamically. This guarantees core content is immediately indexable.

Critical

Blog post pages have no static meta tags — dynamic-only SEO is fragile

The blog-post.html page template has a generic title (Blog | EB Joinery Widnes) and no meta description. The correct title and description are only injected by initBlogPost() after the JS runs. Google will often index the pre-JS state on first crawl, meaning all blog posts share a single undifferentiated meta title. Open Graph tags for social sharing are also entirely absent from the blog post template.

→ Recommended fix

Either generate static HTML files per post, or implement server-side rendering / edge functions (Netlify Edge or similar) to inject the correct meta tags per ?id= parameter before the page is served. At minimum, add unique og:title and og:description meta tags in the JS injection.

Critical

No sitemap.xml or robots.txt referenced anywhere

No sitemap link is present in any page's <head> and no reference to a robots.txt exists. For a 13+ page site with SEO-targeted landing pages (areas-we-cover.html, joinery-services-widnes.html, etc.), a sitemap is essential for ensuring all pages are discovered and crawled efficiently — especially the location-targeted pages which are the primary organic search targets.

→ Recommended fix
<!-- Add to each page <head> --> <link rel="sitemap" type="application/xml" href="/sitemap.xml"> <!-- /robots.txt --> User-agent: * Allow: / Sitemap: https://ebjoinery.co.uk/sitemap.xml
Issue

Portfolio images have no descriptive alt text — just generic "EB Joinery work"

Every portfolio card image is rendered with alt="${p.title}", which resolves to values like "Kitchen Project 3" or "Loft Conversion 1". These are not descriptive to screen readers or search engines. Cloudinary image URLs also don't carry meaningful filenames (they're Facebook CDN URLs with numeric IDs).

→ Recommended fix

Enrich the portfolioItems data with proper alt text strings: e.g. "Bespoke fitted kitchen installation in Widnes — white shaker cabinets with quartz worktop". This also improves accessibility for visually impaired users significantly.

Issue

Open Graph image is the same on all pages — reduces click-through for sharing

Every page uses an identical og:image pointing to a single kitchen photo. When pages are shared to Facebook, WhatsApp, or LinkedIn, they all produce the same preview image regardless of page context. A testimonials page shared to social should ideally show a different image than the contact page.

Issue

No width / height attributes on JS-rendered images — CLS risk

Portfolio and testimonial images rendered via JavaScript have no width or height attributes in the generated HTML (or only generic 600/400 values that may not match actual image dimensions). Without these, the browser cannot reserve layout space before images load, causing Cumulative Layout Shift (CLS) — a Core Web Vitals metric that directly affects Google search ranking.

→ Recommended fix

Set explicit aspect-ratio on image containers in CSS (e.g. aspect-ratio: 4/3) so the browser reserves correct height before the image loads. The existing rule for .portfolio-item img { aspect-ratio: 16/9 } is a good start but needs to be applied to the containing element, not the img itself.

Issue

Blog post URLs use query strings — not ideal for SEO

Blog posts are accessed at blog-post.html?id=1. Google indexes query string URLs, but they are treated as lower-quality than clean path-based URLs (/blog/5-ways-to-add-value-to-your-home). Query string URLs also can't easily be shared as links, are harder to set canonical tags for, and don't benefit from keyword-rich URL slugs.

→ Recommended fix

Migrate to slug-based URLs using Netlify redirects: each post gets a clean slug stored in the post data object, with a Netlify _redirects rule mapping /blog/:slug → /blog-post.html?slug=:slug 200.

Enhancement

BreadcrumbList schema missing from About, Services, and Testimonials pages

About and Testimonials pages have LocalBusiness schema but no BreadcrumbList. Adding breadcrumb schema to all inner pages is a low-effort improvement that consistently helps with SERP appearance, particularly on mobile where Google often shows breadcrumbs instead of URLs.

Enhancement

No <meta name="geo.region"> or hreflang tags for local signals

For hyper-local SEO in the Northwest, adding <meta name="geo.region" content="GB-CHE"> and <meta name="geo.placename" content="Widnes"> reinforces geographic relevance signals. While not a primary ranking factor, they're a quick win for local intent disambiguation.

→ Recommended addition
<meta name="geo.region" content="GB-CHE"> <meta name="geo.placename" content="Widnes, Cheshire"> <meta name="geo.position" content="53.3683;-2.7276">
Summary

Priority Action List

Ordered by estimated impact on conversion and search visibility. Items 1–5 are recommended before the next content update.

  1. Add persistent sticky "Call Now" bar on mobile

    Highest-ROI single change for a local trades site. Fixed bottom bar with tap-to-call and "Get Quote" buttons. Will directly increase phone enquiries from mobile visitors.

  2. Make portfolio overlays visible on touchscreens

    Apply @media (hover: none) to always-show overlays. The gallery is the primary trust-building section — users on phones should see project titles and categories.

  3. Fix mobile menu close-on-link-tap behaviour

    Attach a click listener to all mobile nav links to call classList.remove('open'). A menu that stays open after navigation looks broken.

  4. Make portfolio filter buttons horizontally scrollable on mobile

    Replace wrap with overflow-x: auto; flex-wrap: nowrap at ≤640px. Currently the filters wrap onto 3–4 lines, burying the project grid below the fold.

  5. Add sitemap.xml and reference it in robots.txt and page heads

    Pure SEO hygiene. Without a sitemap, the location-targeted landing pages may take weeks longer to appear in search results. Can be generated statically or via Netlify build plugin.

  6. Reduce inner page hero heights from 100svh to 40–50svh

    Visitors navigating to Portfolio or Testimonials are there to see content immediately, not wait for a full-screen gradient. Reducing the hero height dramatically improves time-to-content on these key pages.

  7. Fix footer gradient direction and white-top contrast failure

    Reverse the footer gradient from to top to to bottom. The current direction produces white at the top of the footer, making white link text invisible on that section.

  8. Pre-render first 12 portfolio items as static HTML

    Ensures portfolio content is immediately indexable by all crawlers. JS can still add more items dynamically — but the core gallery should exist in the page source.

  9. Fix blog-post.html meta tag injection and add OG tags

    Add static fallback meta and implement proper OG tags in the JavaScript post renderer. All blog posts currently share a single undifferentiated meta title on first crawl.

  10. Darken accent colour for on-light-background contexts

    #38bdf8 on white fails WCAG AA contrast. Introduce a --accent-on-light: #0369a1 token for all text used on white/light grey backgrounds.

Overall Assessment

The redesign represents a meaningful step up from a typical trades website. The visual direction is confident, the schema foundation is excellent, and several mobile details (touch targets, iOS zoom prevention) show real quality thinking. The issues identified are largely fixable in a focused sprint — none require a structural rebuild. Addressing the 10 priority items above, particularly the mobile menu bug, portfolio hover-only overlays, missing sitemap, and JS-only portfolio content, would push this site comfortably into the top tier of local joinery websites in the Northwest.
